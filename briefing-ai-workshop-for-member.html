2. Claude vs Claude Code

Claude (대화형 AI)

  • 생각하고 답을 만드는 엔진
  • 파일 업로드 후 요약·분석을 텍스트로 제공
  • 데이터가 크면 샘플링하거나 잘림
  • 중간 과정을 검증하기 어렵다 (블랙박스)
  • 결과를 다시 수작업으로 옮겨야 함

Claude Code (코딩 에이전트)

  • 같은 엔진에 파일을 읽고 코드를 실행하는 손발을 달아준 도구
  • 전체 데이터를 빠짐없이 처리 (샘플링 없음)
  • 코드가 남아 검증·재현 가능 (투명한 과정)
  • 결과를 파일로 저장하고 API로 바로 반영
  • 스킬로 저장해 다음에 한 번에 재실행

같은 Claude 모델을 '채팅 인터페이스'로 쓰느냐, '코딩 에이전트'로 쓰느냐의 차이입니다.

3. AI는 왜 틀릴까: 챙길 것이 다르다

둘 다 '다음에 올 말을 확률로 예측해' 글을 씁니다. 그래서 그럴듯하지만 틀린 답(할루시네이션)이 나옵니다. 뿌리는 같고, 틀렸을 때 드러나는 방식이 다릅니다.

  • Claude (채팅) = 암산 천재, 검산은 안 함. '약 12% 같아요'(실제 9%)처럼 맞는 질문에 틀린 답. → 사람이 검산(숫자가 맞는지)을 챙긴다
  • Claude Code (실행) = 검산은 완벽, 가끔 엉뚱한 걸 계산. 잘못된 전제로 정확한 값을 낸다. → 사람이 전제 확인(무엇을 계산할지)을 챙긴다

한두 번 물어가며 정답에 가까워지는 건 시간 끌기가 아니라, 실행해보고 정보를 모아 추측을 교정하는 과정입니다. 검증 수단(테스트·에러·실제 파일)을 줄수록 빨라집니다.

4. 네 가지 기본 개념: Skill · Agent · API · MCP

카페 운영에 비유하면 이 네 가지로 정리됩니다.

  • Skill · 레시피 = 업무 절차를 적어둔 마크다운 파일 (예: /sync-products)
  • Agent · 매니저 = 레시피 여러 개를 순서대로 묶어 시키는 사람 (예: /run-season-ops)
  • API · 주문 창구 = 외부 서비스와 데이터를 주고받는 정해진 통로 (예: 아임웹 API)
  • MCP · 멀티탭 = API를 Claude Code에 꽂게 해주는 어댑터 (예: Slack MCP)

만든다(Skill) → 묶는다(Agent) → 실행한다(Claude Code) ↔ 연결한다(MCP) → 외부 서비스(API)

5. 더 알아두면 좋은 개념

  • RAG · 검색 증강 = 답하기 전에 내 자료부터 읽기. (지금: MEMORY.md, 노트, 후기 톤 먼저 읽기)
  • Harness · 실행 골격 = 도구와 규칙을 깔아주는 골격. 이 워크스페이스 전체가 하나의 하네스. (지금: 스킬 100개, CLAUDE.md, 메모리)
  • Artifact · 결과물 = 만들어 쌓아두고 다음 작업의 재료로 쓰는 문서·코드·대시보드
  • 카오스 엔지니어링 · 깨질 때를 미리 설계. (지금: 헬스체크 알림, 발송 로그, 멱등 가드)

개념으로 배워서 한 게 아니라, 운영하면서 필요해서 스스로 만들어 온 것들입니다.

6. 스킬 워크숍: 직접 만들어 봅니다

프롬프트는 이렇게 씁니다. ① 내 상황을 최대한 구체적으로 → ② 원하는 결과물 → ③ '이 일을 다시 쓰게 스킬로 만들어줘'

  • 01 API·MCP 없이 웹 크롤링으로 흩어진 기록 모으기
  • 02 API 연결 유튜브 재생목록 → 브리핑 노트 (/analyze-youtube)
  • 03 MCP 연결 회의록 → 할 일 추출 → 구글 캘린더 등록
  • 04 API 연결 글 하나 → 인스타 카드뉴스 → 예약 발행 (/post-to-buffer)

7. 핸즈온: 하루를 기록하는 /run-daily-note

오늘 한 작업을, 어떤 프롬프트로 무슨 결과가 나왔는지까지 데일리 노트 한 장에 남기는 스킬입니다.

# 1) 작업 내역 자동 수집 > 오늘 한 일을 자동으로 모아줘. 대화·만진 파일·git 커밋까지 # 2) 프롬프트·답변·결정·결과 남기기 > daily/오늘날짜.md 에 ①프롬프트 ②답변·처리 ③의사결정 ④결과로 # 3) 스킬로 만들기 > 이 과정을 /run-daily-note 스킬로 만들어줘

부를 때마다 그날 작업이 같은 형식으로 쌓이고, 나중에 펼치면 그날 작업이 재현됩니다.

8. 토큰, 아껴 쓰는 법

토큰 = Claude가 읽고 쓰는 글자 단위. 매 질문마다 지금까지의 대화 전체를 다시 읽습니다.

  • /compact 대화가 길어지면 과거 내용을 요약 압축 (주제 바뀔 때마다)
  • 범위 좁히기 '이 프로젝트 개선해줘' ✗ → 'src/auth.ts의 login 함수 수정해줘' ✓
  • /model sonnet 기본값 Sonnet, 복잡한 작업만 Opus
  • /status 작업 전 세션 정보 확인 · /cost 사용 비용 확인
  • Pro/Max 구독이면 API 비용 없이 사용 가능

9. Gen AI 보안 리스크 관리

  • 공유해도 되는 것 코드·개발 질문, 일반 업무 내용·기획 아이디어, 마케팅 카피·콘텐츠 초안
  • 피해야 하는 것 비밀번호·API 키·인증 토큰, 주민등록번호·계좌·카드번호, 고객 개인정보, 미발표 일정·공개 전 내부 전략

① 민감 정보는 .env로 분리(.gitignore로 업로드 차단) ② 대화는 휘발성, 끝나면 잊힘 ③ Claude Code는 내 컴퓨터에서 실행. 파일은 처리를 위해 Anthropic 서버로 전송되지만 저장되지 않음

10. STAR 프레임워크: 두번째 지능 활용법

S · Start

하고 싶었는데 미뤄왔던 일 → 습관 형성

T · Try

평소 잘 못한다고 생각했던 것 → 도전과 성장

A · Amplify

이미 잘하지만 더 잘하고 싶은 것 → 차별화와 전문성

R · Recover

반복적인 작업을 효율화하는 것 → 효율과 재투자

워크숍: 내 STAR 찾기

  • S: 오랫동안 미뤄온 업무 자동화가 있나요?
  • T: 개발자에게 맡겨야 한다고 생각했던 것은?
  • A: 내가 이미 잘하는 것 + AI = ?
  • R: 반복 업무 중 자동화할 수 있는 것은?

바쁜 직장인 → Start + Recover · 균형 추구 → Start + Try. (X축: 역량 약함→강함, Y축: 효율→성장)

11. 강점차별화 2×2 · BCG 매트릭스

강점차별화 (선호도 × 인정도)

  • Q1 주요 강점 ★ 좋아하고 인정받는 일: 이벤트 기획·운영·진행(100명), 파트너 섭외, 팀 기획, 웹사이트 제작. → 반복 위임, 판단에 집중
  • Q2 전략적 업무 덜 좋아하지만 인정받는 일: 이벤트 기획(25명 이하), CS
  • Q3 축소 고려 덜 좋아하고 덜 인정받는 일: SNS, 슬랙 운영, 영상, 매거진, TF. → 자동화·통합·위임
  • Q4 숨은 강점 좋아하지만 덜 인정받는 일: 중간 서베이, 상세페이지 제작. → 성과를 데이터로 가시화

BCG 매트릭스 (점유율 × 성장률, HFK 운영에 적용)

  • Star ★ 성장 시장의 강한 포지션, 계속 투자: AI부사수 팀, 강점차별화 팀, 스킬 생태계, 100명 이벤트
  • Question Mark 잠재력 있으나 포지션 약함: AI 컨퍼런스, Figma MCP, 다중 컴퓨터 동기화
  • Cash Cow 안정 수익, 효율화로 수확: 정규 성장트랙 팀, 세션 핸드아웃, 상품 운영, 시즌레터
  • Dog 성장도 포지션도 낮음, 축소·통합: SNS 운영, 매거진, 소규모 TF

세 매트릭스 대응: Star ↔ Q1 ↔ Amplify · Question Mark ↔ Q2 ↔ Start · Cash Cow ↔ Q4 ↔ Recover · Dog ↔ Q3 ↔ Try

13. AI = 기본기, HFK = 인사이트와 센스

두번째 지능이란?

  • 첫번째 지능 타고난 인지 능력. 경험·훈련으로 향상되지만 한계가 있고, AI가 빠르게 대체하기 시작한 영역
  • 두번째 지능 AI로 확장된 실행력. 기획자도 개발자의 눈을 가질 수 있고, 미뤄둔 일·엄두 못 냈던 영역에 도전

AI만 있으면 평균 아웃풋이 올라가 차별화가 어려워집니다. AI로 기본기를 빠르게 처리하고, 남는 시간에 인사이트와 센스, 곧 자신만의 암묵지를 기르는 것이 진짜 강점이 됩니다.
